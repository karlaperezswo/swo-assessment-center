Diagrama de Infraestructura

El diagrama de infraestructura es una representación visual completa del entorno actual del cliente y la arquitectura propuesta en AWS. Es un entregable clave que facilita la comprensión del estado actual y el estado futuro.

Objetivo: Crear una representación visual clara y precisa que documente la arquitectura actual y proponga la arquitectura objetivo en AWS, facilitando la comunicación con stakeholders técnicos y de negocio.

Tipos de Diagramas Requeridos

1. Diagrama de Estado Actual (As-Is)

Documenta la infraestructura on-premises existente:

  • Topología de red completa
  • Servidores físicos y virtuales
  • Sistemas de almacenamiento
  • Balanceadores de carga
  • Firewalls y dispositivos de seguridad
  • Conexiones entre componentes
  • Flujos de datos principales
  • Integraciones con sistemas externos

2. Diagrama de Dependencias de Aplicaciones

Muestra las relaciones entre aplicaciones y componentes:

  • Aplicaciones y sus componentes
  • Bases de datos asociadas
  • Servicios compartidos
  • APIs y integraciones
  • Dependencias críticas
  • Flujos de comunicación

3. Diagrama de Arquitectura Propuesta en AWS (To-Be)

Diseño de la arquitectura objetivo en AWS:

  • Regiones y Availability Zones
  • VPCs y subnets
  • Servicios de AWS recomendados (EC2, RDS, S3, etc.)
  • Componentes de seguridad (Security Groups, NACLs, WAF)
  • Balanceadores de carga (ALB, NLB)
  • Servicios de red (Direct Connect, VPN, Transit Gateway)
  • Servicios de monitoreo y logging
  • Estrategia de backup y DR

4. Diagrama de Migración por Waves

Visualización del plan de migración por fases:

  • Agrupación de aplicaciones por wave
  • Secuencia de migración
  • Dependencias entre waves
  • Timeline estimado
  • Quick wins identificados

Herramientas Recomendadas

Para Diagramas Técnicos

Herramienta Uso Recomendado Ventajas
draw.io (diagrams.net) Diagramas generales Gratuito, plantillas AWS, fácil de usar
Lucidchart Diagramas colaborativos Colaboración en tiempo real, integración con AWS
AWS Architecture Icons Iconografía oficial Iconos oficiales actualizados de AWS
Microsoft Visio Diagramas corporativos Estándar corporativo, plantillas profesionales
CloudCraft Arquitecturas AWS 3D Visualización 3D, estimación de costos integrada

Para Mapeo de Dependencias

  • AWS Application Discovery Service: Mapeo automático de dependencias
  • Cloudamize/Concierto/Matilda: Exportación de diagramas de dependencias
  • XMind o MindManager: Mapas mentales de relaciones complejas

Mejores Prácticas

Estándares de Diagramación

  • Consistencia: Usa los mismos iconos y colores en todos los diagramas
  • Claridad: Evita sobrecargar un solo diagrama, crea vistas por capas
  • Leyenda: Incluye siempre una leyenda explicando símbolos y colores
  • Versionado: Mantén control de versiones de los diagramas
  • Escalabilidad: Diseña para que sea fácil actualizar y expandir

Niveles de Detalle

Crea diagramas en múltiples niveles:
  • Nivel 1 - Ejecutivo: Vista de alto nivel para stakeholders de negocio
  • Nivel 2 - Arquitectura: Vista detallada de componentes y servicios
  • Nivel 3 - Técnico: Vista con detalles de configuración y networking

Elementos Esenciales a Incluir

Proceso de Creación

Paso 1: Recopilación de Información

  • Consolidar datos de herramientas de colecta
  • Revisar cuestionario de infraestructura
  • Obtener diagramas existentes del cliente
  • Validar información con equipo técnico del cliente

Paso 2: Diseño del Estado Actual

  • Crear diagrama de topología de red
  • Documentar todos los componentes de infraestructura
  • Mapear dependencias entre aplicaciones
  • Validar con el cliente

Paso 3: Diseño de Arquitectura AWS

  • Aplicar AWS Well-Architected Framework
  • Seleccionar servicios AWS apropiados
  • Diseñar para alta disponibilidad y resiliencia
  • Incorporar mejores prácticas de seguridad
  • Optimizar para costos

Paso 4: Revisión y Validación

  • Revisión interna del equipo SoftwareONE
  • Validación con arquitectos de AWS (si aplica)
  • Presentación al cliente para feedback
  • Iteración basada en comentarios
  • Aprobación final
