Matilda

Matilda es una plataforma especializada en el análisis y migración de sistemas legacy, incluyendo mainframes, AS/400, y aplicaciones monolíticas complejas. Es la herramienta ideal cuando se trabaja con infraestructura heredada.

Prerrequisitos

Requisitos Técnicos

  • Sistemas Operativos Soportados:
    • Windows Server 2008 y superior
    • Linux (RHEL, CentOS, Ubuntu, SUSE)
    • AIX (todas las versiones)
    • Solaris, HP-UX
    • IBM i (AS/400, iSeries)
    • z/OS (Mainframe)
  • Recursos del Sistema:
    • CPU: 2% de uso disponible
    • RAM: 100 MB disponibles
    • Disco: 200 MB de espacio libre
  • Conectividad:
    • Acceso HTTPS saliente al puerto 443
    • URL permitida: *.matilda.cloud
    • Soporte para entornos air-gapped (offline)

Permisos Necesarios

  • Windows: Administrador local
  • Linux/Unix: Usuario root o sudo
  • AIX: Usuario con permisos de sistema
  • AS/400: Perfil con autoridad *ALLOBJ
  • Mainframe: Usuario con permisos de lectura de datasets

Troubleshooting

Problema: Agente no se instala en sistema legacy

Síntomas: Fallo en la instalación del agente en AIX o AS/400

Soluciones:

  1. Verificar compatibilidad de versión del sistema operativo
  2. Confirmar que hay espacio suficiente en disco
  3. Validar permisos del usuario de instalación
  4. Revisar dependencias de librerías del sistema
  5. Consultar logs de instalación en /var/log/matilda/
  6. Contactar soporte especializado de Matilda

Problema: Análisis de código incompleto

Síntomas: No se detectan todas las aplicaciones o módulos

Soluciones:

  1. Verificar que el agente tenga acceso a todos los directorios de aplicaciones
  2. Confirmar que los lenguajes legacy estén soportados (COBOL, RPG, etc.)
  3. Extender el período de análisis para capturar más información
  4. Complementar con análisis manual de código fuente
  5. Revisar configuración de exclusiones en el agente

Problema: Performance degradada en sistema legacy

Síntomas: El sistema se vuelve lento después de instalar el agente

Soluciones:

  1. Ajustar la frecuencia de recolección de métricas
  2. Programar análisis intensivos fuera de horario productivo
  3. Reducir el alcance del análisis de código
  4. Aumentar recursos asignados al sistema (si es posible)
  5. Considerar análisis offline con exportación de datos
